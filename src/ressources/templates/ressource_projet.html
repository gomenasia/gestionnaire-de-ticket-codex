Introduction

Objectif : Développer une application permettant d'analyser et de présenter des statistiques sur les Demandes de valeurs foncières en France, en utilisant des données en Open Data.

L'application doit offrir aux utilisateurs la possibilité de sélectionner différentes granularités (ensemble de villes, départements ou régions) ainsi que des types spécifiques de critères pour obtenir des statistiques pertinentes.

Fonctionnalités

Sélection de Granularité

L'utilisateur doit pouvoir choisir entre 3 granularités principales :

  • Villes — sélectionner un ensemble de villes
  • Départements — sélectionner un ou plusieurs départements
  • Régions — sélectionner une ou plusieurs régions
Sélection de Critères

L'utilisateur doit pouvoir sélectionner un ou plusieurs critères parmi ceux disponibles dans les données, par exemple :

  • Valeur foncière
  • Type de local
  • Surface réelle
  • Surface loi Carrez
  • Surface terrain
  • Nombre de pièces principales
  • etc.

Source des données : https://www.data.gouv.fr/api/1/datasets/r/d573456c-76eb-4276-b91c-e6b9c89d6656

Sélection de Statistiques

L'application doit permettre à l'utilisateur de choisir un ou plusieurs types de statistiques parmi les suivantes :

  • Prix moyens et/ou médians au m² par villes/départements/régions choisis répondant aux critères sélectionnés
  • Distribution des valeurs foncières selon la granularité choisie et les critères sélectionnés
  • Statistiques sur le nombre de pièces principales
  • Statistiques sur la répartition dans l'année des dates de mutation

Pour chaque statistique sélectionnée, l'utilisateur doit pouvoir choisir le type de diagramme souhaité parmi ceux disponibles (Camemberts, diagrammes barre, diagrammes colonnes, etc.).

Génération de Rapports

L'application doit générer des rapports sous forme de pages web contenant les informations et résultats des statistiques sélectionnées. Les rapports doivent être clairs, informatifs et facilement compréhensibles.

Autres fonctionnalités Bonus

Afin de vous démarquer des équipes concurrentes, vous êtes libres d'ajouter de nouvelles fonctionnalités originales à cette application.

Interface Utilisateur (UI)

L'interface utilisateur doit être conviviale et intuitive. Les options de sélection de granularité, de types de critères et de statistiques doivent être présentées de manière claire. L'utilisateur doit pouvoir naviguer facilement à travers l'application.

Contraintes Techniques

L'application doit respecter les contraintes suivantes :

Langage & Bibliothèques
  • Développement en Java obligatoire
  • Utilisation impérative de la bibliothèque visustats développée aux TP précédents pour toutes les visualisations diagrammatiques (Camemberts, diagrammes barre, diagrammes colonnes, etc.)
  • Interface Utilisateur programmée en Java avec le framework Swing
Source de Données

Dans un premier temps, utiliser le fichier de l'année 2024 :

https://www.data.gouv.fr/api/1/datasets/r/af812b0e-a898-4226-8cc8-5a570b257326
  • Format : fichier texte CSV avec le séparateur de champs |
  • Ne télécharger ce fichier qu'une seule fois
  • Tous les tests seront effectués avec un sous-ensemble judicieusement choisi de cette source
  • En production, permettre de sélectionner n'importe quelle année parmi celles disponibles
Note : L'équipe de développement est encouragée à utiliser des outils comme ChatGPT ou Copilot pour accélérer le développement. La contrainte reste que vous devez parfaitement comprendre le code obtenu par ce biais — cela sera évalué en fin de projet.

Mode de Développement & Équipe

Le projet sera réalisé en utilisant une approche agile avec la méthodologie SCRUM. Les itérations seront planifiées, et les retours réguliers seront pris en compte pour améliorer le produit.

Client

L'encadrant de TP joue le rôle de client de cette application.

Product Owner

L'un(e) des étudiant(e)s de l'équipe joue le rôle de Product Owner, responsable de :

  • Définir la vision du produit (en lien avec le client)
  • Prioriser le backlog du produit (liste des fonctionnalités à développer)
  • Prendre des décisions sur les fonctionnalités à implémenter
Scrum Master

Un autre membre de l'équipe joue le rôle de Scrum Master, responsable de :

  • S'assurer que l'équipe suit les principes et pratiques SCRUM
  • Éliminer les obstacles
  • Faciliter les réunions SCRUM
  • Aider l'équipe à atteindre ses objectifs
  • Contribuer activement au développement
Équipe de Développement

L'ensemble de l'équipe est responsable de la réalisation des fonctionnalités définies dans le backlog. Elle est auto-organisée et se fixe des objectifs pour chaque sprint. La qualité du logiciel doit être une priorité dès le début, avec une architecture de tests mise en place dès la première itération.

Planning — Sprints

Sprint 1 Semaines 11–13

Établir les Fondations

  • Mettre en place l'environnement de développement et de test
  • Rédiger les user stories
  • Élaborer l'architecture de base de l'application
  • Implémenter le chargement initial d'un sous-ensemble simplifié des données
  • Mettre en place une interface utilisateur (UI) de base : sélection de granularité et sélection des critères
  • Produire une des statistiques demandées avec un seul type de diagramme possible
  • Rétrospective de l'équipe pour évaluer le processus SCRUM
Sprint 2 Semaines 14–16

Implémentation des fonctionnalités

  • Ajouter à l'UI les fonctionnalités pour configurer les statistiques de base
  • Consolider le package visustats en implémentant les éventuels diagrammes manquants
  • Intégrer la génération de rapports en HTML
  • Rétrospective de l'équipe pour évaluer le processus SCRUM
Sprint 3 Semaines 18–20

Finalisation et Livraison

  • Implémenter le choix du fichier de données
  • Intégration de fonctionnalités additionnelles
  • Terminer l'UI
  • Effectuer des tests finaux et résoudre les problèmes
  • Préparation de la démonstration finale du produit au client
  • Rétrospective de l'équipe pour évaluer le processus SCRUM
Remarques générales :
  • Certaines fonctionnalités plus avancées peuvent être simplifiées ou reportées pour les itérations suivantes.
  • La communication et la collaboration rapides entre les membres seront cruciales pour respecter ce calendrier.
  • La flexibilité est essentielle pour s'adapter aux changements et aux retours d'expérience.
  • 4 séances de TP se feront en autonomie (sans encadrement) — c'est un test du sérieux et de la motivation de l'équipe.

Validation et Acceptation

Validation Interne

L'application sera soumise à une phase de validation interne par l'équipe de développement avant la présentation au client.

Acceptation Client

Phase d'acceptation par le client lors d'une démonstration formelle à la fin du projet.

Évaluation Individuelle

Une évaluation individuelle de performance sera réalisée par l'encadrant de TP la dernière semaine du projet. Il est donc essentiel que chaque membre puisse expliquer et justifier toutes les parties du code, y compris celles obtenues avec l'aide d'outils IA.
