TP 1 — Sujet à réaliser en binôme

Objectif : Réaliser dans VS Code un programme Java qui manipule des formes géométriques. Chaque binôme se concentre sur une forme particulière (Cercle, Ligne, Polygone, Rectangle ou Triangle) implémentant l'interface IForme dans le package fr.univrennes.istic.l2gen.geometrie.

Nota Bene : Il est impératif de respecter le nommage de tous les éléments d'interface : cela sera testé automatiquement et toute erreur entraînera un 0. Les sujets sont volontairement sous-spécifiés — c'est à vous de combler les trous.

TP 3–4 — Système de Gestion de Versions — Git

Objectif : Utiliser Git pour versionner des projets : création de dépôt, récupération, synchronisation à plusieurs, résolution de conflits, et application au projet geometrie.

Se connecter sur le GitLab de l'ISTIC : https://gitlab2.istic.univ-rennes1.fr/

TP 3 — Initiation à Git

3.1 — Création d'un projet et import initial
  1. Sur GitLab : créer un nouveau projet (Create blank project)
  2. Ajouter des participants : Manage > Members
  3. Dans un terminal : git clone <URL>
  4. Ajouter un dossier doc/ avec un fichier README.txt
  5. Versionner : git add, git commit, git push
  6. Vérifier sur GitLab que le dépôt est bien mis à jour
3.5 — Résolution des conflits
  1. Avec votre voisin, modifier la même ligne de deux façons différentes
  2. Propager : git add, git commit, git push
  3. Récupérer : git pull — l'un des deux verra un conflit
  4. Résoudre le conflit puis re-pousser
  5. Répéter pour que le second binôme observe aussi un conflit

5.3 — Création du dossier de test (un seul membre)

5.5 — Écriture des tests unitaires

Pour chaque classe du package, créer une classe de test JUnit. Exemple pour Cercle :

package fr.univrennes.istic.l2gen.geometrie;
import static org.junit.Assert.*;
import org.junit.Before;
import org.junit.Test;

public class CercleTest {
    Cercle c;

    @Before
    public void setUp() {
        c = new Cercle(100, 100, 50);
    }

    @Test
    public void testDescription() { /* ... */ }

    @Test
    public void testDupliquer() { /* ... */ }

    @Test
    public void testEnSVG() { /* ... */ }
}
Important : On ne supprime jamais un test qui passe. Les tests servent de tests de non-régression : ils vérifient que les nouvelles fonctionnalités ne cassent pas l'existant.

TP 7–8 — Test Driven Development

Rappel de la méthode TDD :

  1. Écrire le(s) test(s) montrant que la fonctionnalité X fonctionne
  2. Lancer JUnit et vérifier que le test échoue (X n'existe pas encore)
  3. Écrire le code nécessaire pour passer le test (et pas plus)
  4. Vérifier que le test passe, sinon retourner en 3
  5. Remanier si nécessaire (principe KISS)
